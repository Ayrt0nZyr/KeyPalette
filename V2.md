# Keycap Colorizer — V2 Specification

## Overview

V2 builds directly on the V1 codebase. No structural rewrites — additive features and targeted fixes only. All existing components, hooks, state shape, and data formats from V1 remain in place unless explicitly noted as changed below.

This document is the authoritative build brief for V2. Work through features in the priority order listed. Each feature includes the exact files to touch and the precise behavior expected.

## 1. Fix — Active Key Color Readback

**Problem:** The color picker currently holds whatever color was last picked, regardless of which key is selected. Clicking a key that already has a color does not update the picker to reflect that key's color.

**Fix:** In `useColorway.js`, whenever `selectedKeys` changes to a single key, set `activeColor` to that key's current value in `keyColors`. If multiple keys are selected and they all share the same color, set `activeColor` to that shared color. If multiple keys are selected with differing colors, set `activeColor` to `null` (mixed state — see Polish section).

**Files to touch:**
- `src/hooks/useColorway.js` — update selection logic to sync `activeColor` on key select
- `src/components/Toolbar/ColorPanel.jsx` — handle `activeColor === null` as a mixed/placeholder state; show a muted dashed border on the picker swatch instead of a color

**Behavior:**
- Click key with color `#ff5f57` → picker initializes to `#ff5f57`
- Click key with default color `#e0e0e0` → picker initializes to `#e0e0e0`
- Shift-click to select 3 keys all colored `#1a1a1a` → picker shows `#1a1a1a`
- Shift-click to select keys with mixed colors → picker shows mixed/neutral state

## 2. Fix — Workspace Contrast Tokens

**Problem:** The current light/dark workspace styling is inconsistent. Light keycaps on a light workspace are hard to distinguish. Panel chrome competes with keycap colors for visual attention.

**Fix:** Replace ad-hoc color values with a consistent token system applied via CSS custom properties on `:root` and `[data-workspace="light"]` / `[data-workspace="dark"]`.

**Files to touch:**
- `src/App.module.css` — define all tokens at root level
- Any component CSS module that uses hardcoded background or text colors — replace with tokens

**Dark mode tokens (default):**
```css
--ws-bg: #111111;
--ws-panel: #1e1e1e;
--ws-panel-border: rgba(255, 255, 255, 0.06);
--ws-label: #e8e8e8;
--ws-label-muted: #666666;
--ws-key-border: rgba(255, 255, 255, 0.12);
--ws-input-bg: #2a2a2a;
--ws-input-border: rgba(255, 255, 255, 0.1);
```

**Light mode tokens:**
```css
--ws-bg: #f0f0f0;
--ws-panel: #ffffff;
--ws-panel-border: rgba(0, 0, 0, 0.08);
--ws-label: #1a1a1a;
--ws-label-muted: #999999;
--ws-key-border: rgba(0, 0, 0, 0.15);
--ws-input-bg: #f7f7f7;
--ws-input-border: rgba(0, 0, 0, 0.1);
```

**Key rule:** Every keycap in the SVG must render with a `1px` border using `--ws-key-border` so that light-colored keycaps remain distinguishable against a light workspace background.

## 3. Feature — Reset Button

**Two reset actions, one UI control:**

**"Reset Selected"** — returns all currently selected keys to the default keycap color (`#e0e0e0`). No confirmation required. Appears as a small secondary button in `ColorPanel`.

**"Reset All"** — wipes the entire `keyColors` map back to defaults and resets `caseColor` to `#1a1a1a`. Requires a confirmation prompt (a simple `window.confirm` is acceptable for V2). Appears in the Toolbar, outside `ColorPanel`, clearly separated from per-key controls.

**Files to touch:**
- `src/hooks/useColorway.js` — add `resetSelectedKeys()` and `resetAll()` actions
- `src/components/Toolbar/ColorPanel.jsx` — add "Reset Selected" button
- `src/components/Toolbar/Toolbar.jsx` — add "Reset All" button with confirm guard

**Default keycap color:** `#e0e0e0` (light neutral grey). This is the baseline every key starts at and resets to.

## 4. Feature — Color History Swatch Strip

**What it is:** A horizontal row of up to 16 color swatches showing the most recently used colors, in reverse chronological order (newest left). Persisted in `localStorage` separately from colorway slots.

**Behavior:**
- Every time a color is applied to one or more keys, that hex value is prepended to the history array
- Duplicate hex values are de-duplicated — if `#ff5f57` is already in history, move it to front rather than adding a second copy
- Maximum 16 entries; oldest entries are dropped when the list exceeds 16
- Clicking a swatch loads that color into `activeColor` (same as typing a hex value manually)
- Hovering a swatch shows a tooltip with the hex code
- The strip appears below the `react-colorful` picker in `ColorPanel`

**localStorage key:** `"keycap-colorizer-color-history"` — a plain JSON array of hex strings, separate from `"keycap-colorizer-slots"`

**Files to touch:**
- `src/hooks/useColorway.js` — add `colorHistory` to state, add `pushToHistory(hex)` called whenever a color is applied
- `src/hooks/useLocalStorage.js` — already generic; use it to persist `colorHistory`
- `src/components/Toolbar/ColorPanel.jsx` — render the swatch strip below the picker
- `src/App.module.css` or `Toolbar.module.css` — swatch strip styles

**Swatch dimensions:** 20×20px, rounded corners (`border-radius: 4px`), `1px` border using `--ws-key-border`, arranged in a single wrapping row.

## 5. Feature — Case Finish Toggle

**What it is:** A matte / gloss visual toggle for the keyboard case. The `caseFinish` field already exists in the V1 state shape but is not yet rendered differently.

**Implementation:**

- **Matte:** Case SVG rect renders as a flat fill with `caseColor`. Subtle inner shadow at bottom edge using an SVG `<feDropShadow>` or `filter` to suggest depth.
- **Gloss:** Case SVG rect renders with a thin highlight stroke on the top edge (`rgba(255,255,255,0.18)`, 1px) and a soft linear gradient overlay — white at 8% opacity at top, transparent at 40% down. This simulates a gloss reflection without 3D geometry.

**UI:** A two-option segmented toggle ("Matte / Gloss") placed directly below the case color swatch in the Toolbar. The toggle updates `caseFinish` in state, which `Case.jsx` reads to switch rendering mode.

**Files to touch:**
- `src/components/Keyboard/Case.jsx` — implement matte vs. gloss SVG rendering based on `caseFinish` prop
- `src/components/Toolbar/Toolbar.jsx` — add the segmented toggle UI
- `src/hooks/useColorway.js` — `caseFinish` already in state shape; no change needed

## 7. Feature — Image Palette Extraction Modal

**Replaces** the point-sample eyedropper modal from the original V2 roadmap. Instead of sampling a single pixel (complex cursor UX), this extracts a palette of dominant colors from an uploaded image and presents them as clickable swatches.

**User flow:**
1. User clicks "Extract from image" in the Toolbar
2. A modal opens with a drag-and-drop / click-to-browse file input
3. User uploads any image (PNG, JPG, WEBP)
4. Image is drawn to an offscreen `<canvas>` and sampled to extract 6–8 dominant colors using in-browser color quantization (implement in `extractPalette.js` — no external library)
5. Extracted swatches are displayed in a row inside the modal alongside a thumbnail of the uploaded image
6. User clicks a swatch → `activeColor` is set to that color, modal closes
7. Color applies to whatever keys are in `selectedKeys` via the normal flow

**Files to create:**
- `src/components/ImagePaletteModal/ImagePaletteModal.jsx`
- `src/components/ImagePaletteModal/ImagePaletteModal.module.css`
- `src/utils/extractPalette.js`

**Files to touch:**
- `src/components/Toolbar/Toolbar.jsx` — add "Extract from image" button
- `src/App.jsx` — manage modal open/close state

**`extractPalette.js` signature:**
```js
// Returns array of 6–8 hex strings representing dominant colors
export async function extractPalette(imageFile) { ... }
```

**Implementation approach for `extractPalette`:** Draw the image to a hidden canvas scaled to 100×100px (for performance), sample all pixel data, bin colors into a coarse grid (e.g. 8 levels per channel), rank bins by frequency, return top 6–8 as hex strings after filtering near-duplicates.

**Modal design:** Minimal — image thumbnail on the left (~200px wide), extracted swatch row on the right. Single "Cancel" button at the bottom. No zone-apply shortcuts in V2.

## 8. Feature — PNG Export with Color Legend

**What it is:** Exports the keyboard canvas (case + keys) as a PNG with a color legend appended below, showing all colors used and their hex codes grouped by zone.

**User flow:**
1. User clicks "Export PNG" in the Toolbar or Export panel
2. PNG is generated and downloads automatically — no preview modal in V2
3. Filename: `{colorwayName}-{layout}.png` (e.g. `my-dark-build-tkl.png`), lowercased and hyphenated

**PNG layout (top to bottom):**
- Keyboard canvas (case + keys) at full rendered width
- 16px gap
- Color legend row — for each zone that has keys in the current layout:
  - Zone label text (e.g. "Alphas")
  - One swatch per unique color used in that zone (20×20px)
  - Hex code in monospace below each swatch
- 24px padding on all sides
- Background: current workspace background color (`--ws-bg` value), not transparent

**Implementation:** Use `html2canvas` to capture the keyboard DOM element. Build the legend as an off-screen DOM element, capture it separately, composite both onto a single `<canvas>` using `drawImage`, then call `canvas.toDataURL("image/png")` and trigger download.

**Files to touch / create:**
- `src/utils/exportPng.js` — implement compositing logic (file was stubbed in V1)
- `src/components/ExportPanel/ExportPanel.jsx` — implement (was stubbed in V1)
- `src/components/ExportPanel/ExportPanel.module.css` — implement
- `src/components/Toolbar/Toolbar.jsx` — wire Export PNG button

## 9. Feature — JSON Import / Export

**Export:** Serializes the full current colorway state to a `.json` file and triggers a browser download.

**Exported shape:**
```js
{
  version: 2,
  name: "My Dark Build",
  layout: "tkl",
  caseColor: "#1a1a1a",
  caseFinish: "matte",
  keyColors: {
    "KeyA": "#ffffff",
    "KeyB": "#ff5f57",
    // ...all keys in the layout
  },
  exportedAt: "2026-04-01T12:00:00Z"
}
```

**Import:** User clicks "Import JSON" → native file picker opens (`accept=".json"`) → file is read and parsed → validated → colorway applied.

**Validation checks:**
- `version` field is present and is a number
- `layout` is one of `"60" | "65" | "75" | "tkl" | "full"`
- `keyColors` is a non-null object

On validation failure: show an inline error message inside the Export panel — do not use `window.alert`.

**Filename on export:** `{colorwayName}-{layout}.json`, lowercased and hyphenated.

**Files to touch:**
- `src/utils/exportJson.js` — implement `serializeColorway()` and `deserializeColorway()` (file was stubbed in V1)
- `src/components/ExportPanel/ExportPanel.jsx` — add Import / Export JSON buttons alongside PNG export
- `src/hooks/useColorway.js` — add `importColorway(data)` action

## 10. Polish — Hex Badge & Mixed-Color Indicator

Implement these last, after all features above are working.

### Hex Badge on Canvas

When exactly one key is selected, render a small floating badge centered just below that key on the SVG canvas displaying its current hex color (e.g. `#ff5f57`).

- Clicking the badge copies the hex value to clipboard via `navigator.clipboard.writeText`
- Badge style: `10px` monospace font, pill shape (`border-radius: 10px`), background `var(--ws-panel)`, border `1px solid var(--ws-panel-border)`, `4px` gap below the key
- Only visible when `selectedKeys.length === 1`
- Position: absolutely placed relative to the SVG, centered on the key's x-midpoint

**Files to touch:**
- `src/components/Keyboard/Keyboard.jsx` — render badge conditionally based on `selectedKeys`
- `src/components/Keyboard/Keyboard.module.css` — badge styles

### Mixed-Color Indicator

When multiple keys with differing colors are selected (`activeColor === null`):

- The color picker swatch area shows a grey/white diagonal stripe pattern (CSS `repeating-linear-gradient`) to indicate "no single color"
- A label reads "Multiple colors — pick to override all" in `var(--ws-label-muted)`
- As soon as the user picks any color, it applies uniformly to all selected keys and the mixed state clears

**Files to touch:**
- `src/components/Toolbar/ColorPanel.jsx` — conditional rendering based on `activeColor === null`

## State Shape Changes for V2

One addition to the V1 state shape in `useColorway.js`:

```js
{
  // All V1 fields unchanged:
  layout: "tkl",
  caseColor: "#1a1a1a",
  caseFinish: "matte",
  keyColors: { ... },
  selectedKeys: [],
  activeColor: "#ffffff",       // null when mixed selection
  colorwayName: "My Colorway",
  workspace: "dark",

  // New in V2:
  colorHistory: [],             // up to 16 hex strings, newest first
}
```

`colorHistory` is persisted via a separate localStorage key (`"keycap-colorizer-color-history"`), not inside the colorway slot objects.
